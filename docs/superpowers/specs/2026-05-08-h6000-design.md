# H6000 — Performance Sailing Instrument Processor

**Date:** 2026-05-08
**Status:** Design approved, ready for implementation planning
**Working name:** H6000 (one notch above the B&G H5000 it replaces)

## 1. Overview

A custom embedded instrument processor for a performance catamaran. Replaces the
brain (CPU) of an existing B&G H5000 system while keeping the H5000 course
computer (rudder drive amplifier) and existing transducers in place. Equivalent
in role to a B&G H5000 CPU or a Pixel Surfer Exocet.

**Primary functions:**

- Aggregate inputs from NMEA 2000, NMEA 0183, an IMU, and analog sources.
- Compute calibrated true wind, polar performance, leeway, current, and laylines.
- Emit calibrated values back onto the N2K bus so the existing Zeus SR plotter
  and any other displays show better data without modification.
- Drive the H5000 course computer for autopilot, replacing the H5000 CPU's
  autopilot logic.
- Host a local web interface for configuration, calibration wizards, and (in
  later phases) live helm dashboards.

**Personal build for owner's catamaran.** No NMEA 2000 certification, no IP67
enclosure design, no commercial scope concerns. Optimize for capability and
hackability over BOM cost.

## 2. Scope & phasing

| Phase | Goal | Autopilot involvement |
| --- | --- | --- |
| **0a** | Sensor stack on prototype hardware (CM5 + dongles). True-wind, polars, calibration. Emit calibrated PGNs onto bus. | Listen-only — decode and display autopilot PGNs |
| **0b** | Shadow autopilot — compute commands, write to session log, do not transmit. Diff against H5000 traffic. | Shadow only |
| **0c** | Live autopilot — primary/standby switch in UI, watchdog, sanity bounds. Take over from H5000. | Primary, with H5000 as hot fallback |
| **1**  | Custom MCU board replaces dongles. Same software stack. | Unchanged |
| **2**  | Helm dashboard for tablet/phone (configurable big-number tiles, polar plot, layline overlay). | Adds UI controls |
| **3**  | Dedicated cockpit displays. | — |

Phase 0 is the focus of this spec. Phases 1–3 are sketched only to show the
architecture supports them.

### Out of scope

- Race-tactics features: start-line, race timer, mark roundings.
- Cellular telemetry, fleet tracking, cloud sync.
- NMEA 2000 certification, EMC compliance.
- Custom course computer / rudder drive amplifier (B&G's stays).
- Production manufacturability — single unit for owner's boat.

## 3. Hardware

### 3.1 Phase 0 BOM (~$400 if Pi not already owned)

- **Compute:** Raspberry Pi CM5 in CM5 IO board (or RPi 5). 4–8 GB RAM, 32 GB
  eMMC, NVMe slot for log storage.
- **N2K interface:** Actisense NGT-1-USB. Well-documented Actisense binary
  protocol; canboat-compatible.
- **NMEA 0183:** 2× USB-RS422 dongles (FTDI-based, e.g. Plugable / Brainboxes).
- **IMU:** BNO085 breakout (Adafruit / Sparkfun). Fused quaternion + Euler
  output via I²C.
- **Power:** 12 V boat → 5 V/USB-C PD step-down with isolated input, ≥3 A.
- **Network:** built-in WiFi (client mode on boat WiFi; AP mode fallback) and
  Ethernet (shoreside dev).

### 3.2 Phase 1 custom board (sketched)

Replaces all dongles with a single STM32H7-class MCU board:

- 1× CAN-FD transceiver with N2K-compatible biasing
- 4× RS-422 transceivers on UARTs (galvanically isolated)
- BNO085 on SPI (faster than I²C)
- 24-bit external ADC (e.g. ADS1256) on SPI for load pins / strain gauges
- Built-in 16-bit ADC for low-rate analog
- USB-C device port (CDC for data, DFU for flashing)
- 9–32 V isolated input
- Same enclosure footprint as the SBC carrier

CM5 keeps running unchanged; only the bridge daemon's wire driver swaps from
"NGT-1 + USB-RS422 + I²C BNO085" to "single MCU over USB-CDC".

## 4. Software architecture

### 4.1 Process & language

Single Node.js process running TypeScript. Same stack family as the rest of the
owner's workspace (Next.js, Drizzle, SQLite). One systemd service.

```
autopilot/
├── packages/
│   ├── core/        shared types, codecs, math helpers
│   ├── bridge/      daemon: serial I/O, NGT-1, IMU, 0183, persistence
│   ├── compute/     true wind, polars, leeway, laylines
│   └── web/         Next.js UI + SSE route + Drizzle config
└── apps/
    └── autopilot-server/  composes everything in one process
```

npm workspaces. Each package independently testable.

### 4.2 Pub/sub bus

In-process, RxJS-based:

- `bus.publish(sample)` fans out to subscribers of the channel.
- `bus.subscribe(pattern, handler)` supports exact (`wind.apparent.angle`) and
  wildcard (`wind.**`) subscriptions.
- Backpressure: bounded ring buffer per subscriber, oldest drops with a
  warning. One slow subscriber cannot stall the bus.

No external broker. A Signal K-compliant HTTP endpoint can be layered on later
without changing the internal bus.

### 4.3 Data model

```ts
type Sample = {
  channel: string;          // hierarchical: "wind.apparent.angle"
  t_ns: bigint;             // ns since Unix epoch
  value: ChannelValue;      // typed: number | Vec3 | Quaternion | GeoPoint | enum
  source: SourceTag;        // "n2k:127250@dev0x10" | "0183:port1" | "imu" | "computed:true_wind"
};
```

- Hierarchical channel names with dots: `wind.apparent.*`, `wind.true.*`,
  `wind.true.calibrated.*`, `nav.gps.*`, `boat.heading.{magnetic,true}`,
  `motion.{heel,pitch,yawRate}`, `autopilot.{mode,target.heading,target.twa}`.
- BigInt timestamps; downcast only when interpolating tables.
- Source tagging is mandatory — used for selector arbitration and debugging.
- No "current state" object; subscribers compute their own state from the
  stream. Keeps the bus replayable.

This shape matches Signal K closely; converting to a Signal K HTTP endpoint
later is a thin adapter.

### 4.4 Bridge architecture (3 layers)

The wire driver is the only layer that changes between Phase 0 and Phase 1.

```
              ┌────────────────────────────────────────┐
              │ Channel mapper                         │ ← Phase-stable
              │ (PGN field → "wind.apparent.angle")    │
              └────────────────────────────────────────┘
                              ▲ ▼ Sample
              ┌────────────────────────────────────────┐
              │ Decoder layer                          │ ← Phase-stable
              │ • N2K PGN decode (canboat database)    │
              │ • Fast-packet reassembly (always)      │
              │ • 0183 sentence parser                 │
              │ • IMU sample shape                     │
              └────────────────────────────────────────┘
                              ▲ ▼ Frame / sentence / sample
              ┌────────────────────────────────────────┐
              │ Wire driver (pluggable)                │ ← Phase 0 vs Phase 1
              │ Phase 0: NGT-1 binary + USB-RS422 +    │
              │          I²C/USB BNO085                │
              │ Phase 1: Custom MCU over USB-CDC       │
              └────────────────────────────────────────┘
                              ▲ ▼ Bytes
                            Hardware
```

**Critical decision:** the Phase 0 NGT-1 driver emits raw CAN frames, even
though NGT-1 also offers a higher-level pre-decoded mode. Reassembling
fast-packet PGNs always happens in the decoder layer. This keeps the contract
honest for Phase 1, where the MCU emits raw CAN frames natively.

**Channel mapping is in TypeScript** (not YAML/JSON). Typed, refactorable,
hot-reload makes the recompile cost moot. A reverse map is provided for
TX (channel → outbound PGN encoder).

### 4.5 Wire driver interface

```ts
interface WireDriver {
  rxCan: Observable<RawCanFrame>;     // { id, ext, data, rxTimestamp }
  rx0183: Observable<Raw0183Line>;    // { port, bytes, rxTimestamp }
  rxImu:  Observable<ImuSample>;      // { quat, gyro, accel, t }
  rxAdc:  Observable<AdcSample>;      // { channel, value, t }
  txCan(frame: RawCanFrame): Promise<void>;
  tx0183(port: number, bytes: Buffer): Promise<void>;
  health: Observable<DriverHealth>;
}
```

### 4.6 Phase 1 MCU↔SBC frame format (sketched)

When the custom board lands:

- COBS-encoded length-delimited frames on USB-CDC.
- Tagged-union messages, single-byte tag + payload, with a leading version
  byte.
- Tags: `RX_CAN`, `RX_0183`, `RX_IMU`, `RX_ADC`, `LOG`, `TX_CAN_REQ`,
  `TX_0183_REQ`, `CFG_GET`, `CFG_SET`.
- Every RX message carries the MCU's monotonic ns timestamp at capture.
  Bridge tracks ns offset to Unix-ns via `CFG_GET` round-trip RTT.

Not built in Phase 0. Listed here so the Phase 0 wire driver doesn't accidentally
couple to NGT-1 quirks.

### 4.7 Source priority & arbitration

When two devices emit the same channel (e.g. GPS over N2K and over 0183), a
"selector" subscribes to all sources for that channel and emits the
highest-priority source whose last sample is younger than a per-channel
freshness window (default 2 s, configurable). Priorities and freshness windows
are configured in `config.db` per channel; sensible defaults (N2K > 0183 >
computed) plus user override. The freshness window here is independent of the
compute-pipeline staleness guards in §6.1 — selectors gate which source is
*chosen*; pipelines gate when a compute tick *runs at all*.

## 5. Persistence & replay

Two distinct stores:

| Store | Contents | Format | Why |
| --- | --- | --- | --- |
| `config.db` | Boat config, polars, cal tables, autopilot settings, source priorities, user prefs | SQLite via Drizzle | Small, durable, edited from web UI |
| Session log | Raw inbound bytes from each input, timestamped | `.jsonl.gz`, one file per session | Perfect-fidelity replay, post-mortem, calibration tuning |

Session log line format:

```
{"t_ns":..., "src":"ngt1", "kind":"raw", "bytes":"7e0a..."}
{"t_ns":..., "src":"0183:port1", "kind":"raw", "bytes":"$GPGGA,..."}
{"t_ns":..., "src":"imu", "kind":"sample", "value":{...}}
```

Replay reads the file and feeds it through the same decoder layer. The whole
stack (compute, N2K transmit) runs identically against replayed sessions, with
TX sandboxed to a write-to-file sink instead of the bus.

## 6. Compute model

### 6.1 True-wind pipeline

Inputs subscribed (with staleness guards — drop if any input >2 s old):

- `wind.apparent.{angle,speed}` (AWA, AWS) — masthead
- `boat.speed.water` (BSP)
- `boat.heading.{magnetic,true}` (HDG)
- `motion.{heel,pitch,yawRate,pitchRate,rollRate}`
- `nav.gps.{cog,sog}`
- (future) daggerboard position, hull trim — Phase 1+

Pipeline stages (pure functions in `packages/compute/src/trueWind/`):

```
raw AWA, AWS  ─┐
               ├─► [1] masthead-motion correction       ─► AWA', AWS'
IMU rates ─────┘    (subtract masthead linear velocity:
                     yaw_rate · mast_height + pitch_rate
                     contributions in masthead frame)

AWA', AWS'  ───────► [2] calibration table lookup       ─► AWA_cal, AWS_cal
                     (bilinear interp on 2D grid
                      indexed by (AWS bin, AWA bin))

AWA_cal, AWS_cal ──┐
BSP_cal           ─┤► [3] vector subtraction            ─► TWS, TWA, TWD
HDG_true          ─┘   (boat frame, then rotate to earth)
```

`BSP_cal` from a 1D BSP correction table. `HDG_true = HDG_mag + magvar` corrected
by a 1D compass deviation table (36 bins × 10°).

### 6.2 Calibration tables (three, all editable from UI, hot-reloaded)

| Table | Shape | Filled by |
| --- | --- | --- |
| AWS/AWA correction | 2D, ~8×12 grid (AWS bins × AWA bins) | Tack-test wizard |
| BSP correction | 1D, ~10 speed bins | GPS-correlated runs in still water |
| Compass deviation | 1D, 36 bins (10° each) | Dockside swing or COG correlation while sailing |

**Tack-test wizard** logic: on a tack, true wind direction must be the same as
on the previous tack. Differences in computed TWD reflect wrong AWS/AWA
correction at the AWA sailed. The wizard captures steady runs on each tack,
adjusts the relevant cells of the 2D grid until both tacks agree, repeats at
varied wind speeds.

**Catamaran specifics:**

- Heel correction (which dominates monohull math) is mostly noise on a cat
  (heel <8°). Skip the heel dimension initially; add later if data shows it
  matters.
- Hull lift and foil-mode behavior are deferred to Phase 1+. A `mode` channel
  (`hull` / `foil`) selects between alternative polar/cal tables when added.

### 6.3 Polars

```
polars[TWS_bin][TWA_bin] → target_BSP

Outputs:
  performance.target.boatSpeed     interp(polars, TWS, |TWA|)
  performance.percentPolar         BSP_actual / target_BSP × 100
  performance.target.twaUpwind     argmax_TWA(VMG_upwind(polars, TWS))    (cached)
  performance.target.twaDownwind   argmax_TWA(VMG_downwind(polars, TWS))  (cached)
  performance.vmg                  BSP × cos(TWA)
  performance.targetVmg            target_BSP(target_TWA) × cos(target_TWA)
```

Source format: **Expedition CSV** (`tws,twa,bsp` rows). Importable from the UI;
per-cell editable.

`argmax_TWA` solves are computed once at table-load time, cached per TWS bin.

### 6.4 Leeway & current — Option C (hybrid)

Three components running together:

1. **Physics baseline (always on):**
   ```
   leeway_baseline = K_cat(board_config) · AWS²·CL_lat(AWA) / BSP²
   ```
   Configurable `K_cat` per board configuration (down/half/up).

2. **Observation logging (always on, when steady):**
   When sailing is "steady" (no large rate-of-turn for >30 s, no big AWS
   change), record:
   ```
   leeway_observed ≈ HDG_true − COG − current_set_drift_correction
   ```
   per (TWS bin, TWA sign, board config) cell. Cells average over time.

3. **Two-state Kalman-ish filter** (current/leeway separator):
   ```
   state = [current_set, current_drift, leeway_K]    (slow & fast)
   observation = (COG, SOG) − rotate(HDG, BSP)       water/ground residual
              ≈ current_vector + (BSP·sin(leeway)) · boat_lateral_unit
   ```
   Process noise much larger on `leeway_K` than on `current_*` — separation
   by time constant.

4. **Live blend:** `leeway_live = blend(baseline, table_cell)` where blend
   weight ramps with the cell's observation count. Sparse cells stay model-only.

5. **Foil-mode switch:** a `mode` channel (manual button in UI, later auto)
   selects between hull and foil tables.

### 6.5 Laylines

Given current waypoint (from chart plotter nav PGN, or set in UI), project
layline rays at `target_TWA_upwind` from the mark, account for current vector,
project onto a chart in the UI. Time-to-layline computed alongside.

### 6.6 Update rates

Driven by slowest input — no scheduler:

- True wind: 5–10 Hz (matches AWA/AWS broadcast rate)
- Polars: piggyback on true wind
- Current: low-passed, ~0.1 Hz visible
- Laylines: 1 Hz

## 7. N2K & autopilot integration

### 7.1 PGNs received (decoded → bus)

Standard sailing/nav set:

| PGN | Channels |
| --- | --- |
| 130306 | `wind.apparent.{angle,speed}`, raw `wind.true.*` (pre-cal) |
| 128259 | `boat.speed.water` |
| 128267 | `nav.depth` |
| 127250 | `boat.heading.{magnetic,true}` |
| 127257 | `motion.{heel,pitch,yaw}` |
| 127245 | `boat.rudder.angle` |
| 129025/26/29 | `nav.gps.{position,cog,sog,satellites,fix}` |
| 129283/84/85 | `nav.{xte,navData,routeWaypointInfo}` |
| 128275 | `nav.distance.log` |
| 130310/11/12 | `env.{water.temp,air.temp,humidity,baro}` |
| 127488/89/93 | `engine.*`, `fuel.*` |
| 127505/8 | `tank.fluidLevel`, `battery.*` |
| 127237 | `autopilot.{mode,target.heading,target.twa,actual.heading}` (standard) |

**Proprietary B&G PGNs** (manufacturer code 1857) for the H5000 ↔

- 65302, 65305, 65360, 65371, 65379, 65384 (canboat-documented autopilot
  control & status)
- 130845, 130850 (performance/event commands the H5000 emits to the course
  computer)

We start from the canboat JSON PGN database. Proprietary PGNs are decode-only
in Phase 0a/0b; their semantics are confirmed against captured boat traffic
before we transmit them in 0c.

### 7.2 PGNs transmitted

| PGN | Source | When |
| --- | --- | --- |
| 130306 (true wind) | calibrated true-wind pipeline | always, 5–10 Hz |
| 127237 (heading/track control) | autopilot pipeline | only when primary mode |
| Proprietary 1857 PGNs | autopilot pipeline | only when primary, only those needed by the course computer |
| 130823+ proprietary (us) | performance pipeline | optional, for our own future displays |

All TX gated by `mode === primary && healthy`.

### 7.3 Autopilot mode arbitration

```
mode = primary | standby | shadow
  primary  : I transmit autopilot commands
  standby  : I receive only; I do not transmit autopilot commands
  shadow   : I compute commands and log them; I do NOT transmit
             (Phase 0b mode — for parity diffing against the H5000)
```

**Single-transmitter invariant:** at most one box has `mode=primary` at any
instant. Switching is explicit and human-initiated; no auto-failover to/from
the H5000.

**Switching mechanism:** software-only. Web UI button arms/engages, persists in
`config.db`. Default = `standby` on boot. The owner manually puts the H5000 CPU
in standby when transferring authority to our box, and vice versa.

### 7.4 Safety model

Autopilot pipeline runs through a safety gate before any TX:

1. **Compute watchdog:** compute tick must arrive every <500 ms. Miss → zero
   TX, raise `alarm.autopilot.watchdog`.
2. **Output sanity bounds:** `|target_heading_change_per_tick| < 5°`,
   `|rudder_command| < 30°`, mode in legal set. Out-of-bounds → zero TX, raise
   alarm.
3. **Two-step engagement** in UI: arm → engage. Default = disarmed on boot.
4. **Heartbeat PGN at 1 Hz** containing `{mode, healthy, watchdog_age}`.
   Liveness for our own UI and for the H5000's awareness.
5. **Local-network only** for command-issuing clients. No remote engagement.
6. **Input-side staleness guards** on every autopilot input: if HDG hasn't
   updated in 500 ms, or HDG rate-of-change exceeds physically possible, the
   autopilot pipeline disengages itself and posts an alarm. *Sensor* bad data
   is the most common autopilot incident.

No hardware E-stop. No physical 3-position helm switch. No automatic failover.

### 7.5 Phase 0b shadow mode

Our box runs the autopilot pipeline with `mode=shadow`: same inputs, same
compute, generates the same PGNs we would transmit, but writes them to the
session log instead of the bus. A diff tool compares our shadow TX against the
H5000's actual TX in the same session log.

Shadow mode is a tool, not a hard gate for going live. The owner uses
shadow-vs-actual parity to decide when confidence is high enough to switch
`mode → primary` for the first time.

## 8. Web UI

### 8.1 Stack

- Next.js (App Router) + TypeScript + Tailwind. Same as the rest of the owner's
  workspace.
- No `next-auth` — Phase 0 binds to LAN only, optional shared password gate via
  middleware.
- Live data over **SSE** (Server-Sent Events). One-way 5–10 Hz updates fit SSE
  cleanly; no WebSocket complexity, no custom server.
- Commands to the box are plain POSTs.
- Drizzle + better-sqlite3 for `config.db`.

### 8.2 Phase 0 page inventory

| Route | Purpose |
| --- | --- |
| `/` | System status: bridge health, MCU/dongle connection, bus utilization, source list, watchdog age |
| `/inspect` | Live channel inspector — every channel, current value, source, last-update age, sparkline |
| `/calibration/wind` | Tack-test wizard for AWS/AWA cal table, live preview of cal effect |
| `/calibration/bsp` | GPS-correlated BSP cal |
| `/calibration/compass` | Compass deviation cal |
| `/polars` | Polar editor: Expedition CSV import, table edit, polar curve plot |
| `/boat` | Boat config: rig dimensions, sensor mounting offsets, masthead height, daggerboard config |
| `/autopilot` | Engage/disengage, mode select, target heading/TWA, live status |
| `/sessions` | Browse session logs, download, replay-into-sandbox |
| `/n2k` | Raw N2K traffic monitor: PGN rates, errors, devices on bus |

### 8.3 Phase 2 helm dashboard (sketched)

Separate route group `/helm/*` with sunlight-readable layouts, configurable
big-number tiles, polar plot, layline overlay on a small chart. Same SSE feed,
different UI. Out of scope for Phase 0.

### 8.4 Live data path

```
Bus (RxJS Subjects)
  → SSE route handler subscribes by channel pattern
  → throttled writes (100–200 ms aggregation)
  → res.write("data: {...}\n\n")
  → browser EventSource per page
  → rAF-batched React state setter
```

Throttle on both sides: don't fan out 100 Hz to browsers; rAF-batch on browser
to avoid render thrash. Payload is `{ channel, t, value }` only.

### 8.5 Configuration round-trips

UI edits write to `config.db` (Drizzle), and a `config-changed` event publishes
on the bus. Compute pipelines subscribe and hot-reload affected tables. No
restart. This is what makes the tack-test wizard interactive — tweak a cell,
see corrected TWD update live.

## 9. Testing strategy

Three layers, in order of value:

1. **Pure-function unit tests** in `core` and `compute`. Codecs, PGN decoders,
   true-wind math, polar interpolation, leeway model, Kalman filter. Property
   tests for round-trips: feed AW + boat motion → recover same TW; or
   `AW = TW + boat_motion` identity.
2. **Replay tests on real-data fixtures.** A handful of representative sessions
   captured from the boat (light air, breeze, downwind run, autopilot-engaged)
   saved as `.jsonl.gz` test fixtures. Test = "replay session N, assert
   derived channels stay sane, no `NaN`, no panics, TWD doesn't jump >X°/s
   except across a tack." Bugs encountered on the boat get added to the
   fixture set as regression tests.
3. **Integration test against virtual NGT-1.** A test driver emits canned
   Actisense binary frames into a loopback serial pair; the bridge reads it as
   if from real hardware; the test asserts SSE output matches expectations.
   Used in CI before real hardware is available.

End-to-end-with-hardware tests are manual; checklist lives in `docs/`.

## 10. Build sequence (Phase 0)

Linear-ish, optimized for "earliest moment on the boat with useful data":

1. Workspace skeleton (`packages/core`, `packages/bridge`, `packages/compute`,
   `packages/web`, `apps/autopilot-server`).
2. `core` types: `Sample`, `Channel`, `Bus` with RxJS, channel-name constants.
3. NGT-1 driver: open `/dev/ttyUSB0`, parse Actisense binary, emit raw CAN
   frames.
4. N2K decoder: canboat PGN database integration, fast-packet reassembly,
   decode → channel mapper (TypeScript-coded).
5. Bridge wires #3 → #4 → bus. **First milestone: real PGN values visible on
   the bus.**
6. 0183 input driver + decoder.
7. IMU driver (BNO085 over I²C or USB).
8. Persistence: `.jsonl.gz` session logger.
9. Replay driver (reads `.jsonl.gz`, emits onto bus identically to live).
10. Next.js skeleton + SSE route + `/inspect` page (live channel inspector).
11. SQLite + Drizzle, `config.db`, boat config page.
12. True-wind compute pipeline (without cal — identity correction first).
13. AWS/AWA cal table editor + tack-test wizard.
14. Polars: Expedition CSV parser, table editor, target-speed compute.
15. N2K transmit path: encode our calibrated true wind PGN 130306, push back
    to bus.
16. **Take to the boat — first useful run.** Capture sessions for offline
    tuning.
17. BSP cal, compass deviation, leeway observation logging.
18. Autopilot pipeline: receive H5000-CPU-to-course-computer PGNs, decode,
    display in `/autopilot`.
19. Shadow-mode autopilot pipeline (compute commands, log only, don't
    transmit).
20. Hybrid leeway model (Option C): physics baseline + observation table +
    Kalman current/leeway separator.
21. Autopilot live engagement (Phase 0c): primary/standby in UI, watchdog,
    sanity bounds, then go live in calm conditions.

## 11. Open questions for implementation phase

These are deferred to the implementation plan, not design-blocking:

- Exact canboat-ts library choice and how it integrates with our raw-CAN model.
- BNO085 connection method (I²C off Pi GPIO vs. dedicated USB breakout).
- 0183 port count needed for the actual boat — start with 2, expand if traffic
  demands.
- Whether the compute pipeline runs as part of the main Node process or as a
  worker thread (perf-driven decision, made after first benchmarks).
- Polars source: owner-supplied CSV vs. starting from a generic cat polar
  template.
- Boat WiFi configuration approach: client-only, AP fallback, or both.
- Cell binning strategy for the leeway lookup table (TWS bin width, TWA
  bucket count).
