# Chart right-click context menu for route editing

**Date:** 2026-05-30
**Status:** design (approved in brainstorming)
**Related:** #21 (leg-by-leg routing, route connector), #23 (chart-based route building / drag-edit — future)

## Summary

Add a right-click (`contextmenu`) menu to the `/chart` MapLibre map for editing the
in-progress route directly on the chart, instead of only through the plan panel's
dropdowns. The menu is **context-sensitive** to what's under the cursor:

- **A waypoint marker** → actions reference that specific waypoint by name
  ("Add **Newport** to route", "Remove **Newport** from route", "Set **Newport**
  as start / destination", "Delete **Newport**").
- **A route-connector leg** → "Insert waypoint here" (creates a saved waypoint at
  the click point and inserts it into the sequence at that leg).
- **Empty water** → "Add waypoint here", "Route to here", "Clear route".

New points are created as **saved waypoints** (one data model — see brainstorming
decision). This unifies the in-progress route into a single ordered list of
waypoint IDs shared by the panel and the menu.

## Non-goals (v1)

- **Touch long-press menu.** v1 is right-click (desktop / trackpad two-finger).
  The Map already has long-press infra (`suppressLongPressLayers`); wiring a touch
  long-press to the same menu is a clean follow-up, not v1.
- **Drag-to-edit route handles** (#23). Waypoint *marker* drag already exists
  (`WaypointsLayer.onMoveWaypoint`); dragging route *legs* to insert is out of scope.
- **Reorder-by-drag** of the via list. v1 reorders via insert/remove only
  (the panel keeps its ↑/↓ buttons).
- **Editing the wind-optimized path.** The menu edits the *route* (waypoints); the
  optimized path is regenerated by pressing Plan, unchanged.
- **Multiple *simultaneous* routes on the chart.** v1 edits a single **active**
  route (`routeWaypointIds`). Multiplicity is already covered by the saved-routes
  catalogue (the `routes` table / `/routes` page / saved-route mode): build/edit
  one active route, optionally save it, load a different one. Drawing several
  routes at once with an active-route switcher is a larger follow-on (a
  route-manager) and would change the core single-list data model — explicitly out
  of v1. Keeping one active route is what makes the unified `routeWaypointIds`
  model clean.
- **"Save route as…" from the chart.** Now that the active route is just an ordered
  `waypointIds` list (identical to the saved-route shape), persisting it via
  `POST /api/routes` is trivial — but it's a small near-term *follow-up*, not part
  of this menu. Saving stays in the existing `/routes` flow for v1.

## 1. Unified route state (`useRoutePlan`)

Today the in-progress route is split: `routeStartId` / `routeEndId` live in
`chart/page.tsx` (605–606), the editable `viaIds` lives inside `RoutePlanPanel`.
A map-driven editor needs one shared source of truth.

Introduce a `useRoutePlan()` hook (called once in `page.tsx`) holding:

```ts
routeWaypointIds: string[]   // ordered: [start, ...via, end]
```

with derived `start = ids[0]`, `end = ids[ids.length - 1]`, `via = ids.slice(1, -1)`,
and pure mutators (each returns a new array; unit-tested in isolation):

- `append(id)` — add to the end (becomes the new destination).
- `remove(id)` — remove the first occurrence.
- `insertAt(index, id)` — insert at a position (for leg "insert here").
- `setStart(id)` — make `id` the first element (move if already present, else prepend).
- `setEnd(id)` — make `id` the last element (move if present, else append).
- `setIds(ids)` — replace wholesale (saved-route selection).
- `clear()` — empty the route.

The mutators live in a pure module (`lib/route-plan.ts`) so they're testable without
React; the hook wraps them around `useState`.

State is ephemeral (not persisted), matching the current "start/end deliberately
omitted from persistence" decision in `page.tsx`. `routes` (the optimized output)
persistence is unchanged.

## 2. Wiring the connector + panel to the shared state

- **Connector:** `page.tsx` derives `routeWaypointPath` directly from
  `routeWaypointIds` resolved against `waypoints` (so the existing
  `attachRouteConnector` effect feeds from the shared state). The
  `onWaypointPath` prop added to `RoutePlanPanel` in #21 is **removed** — page.tsx
  now owns the derivation, a simplification.
- **Panel** (`RoutePlanPanel`) becomes a controlled view of the shared state:
  - Pick-waypoints mode: Start dropdown → `setStart`, End dropdown → `setEnd`,
    the via list → `insertAt` / `remove` / reorder over the middle ids.
  - Saved-route mode: selecting a route → `setIds(route.waypointIds)`.
  - `mode` and `routeId` stay panel-local (UI state). The panel's internal
    `viaIds` / `startId` / `endId` are removed in favor of the shared ids.
- `routeStartId` / `routeEndId` in `page.tsx` are replaced by the derived
  start/end (used for the green/red endpoint mark badges, lines ~856).

## 3. Context menu component (`ChartContextMenu`)

A cursor-positioned popover rendered in `page.tsx`. Props:

```ts
{
  open: boolean;
  screen: { x: number; y: number };       // where to render
  target:
    | { kind: 'empty'; lat: number; lon: number }
    | { kind: 'waypoint'; waypoint: Wp; inRoute: boolean }
    | { kind: 'leg'; lat: number; lon: number; insertIndex: number };
  onClose: () => void;
  // action callbacks (wired to mutators + waypoint create/delete) ...
}
```

Items by `target.kind`:

| Target | Items |
|--------|-------|
| `waypoint` (not in route) | Add **{name}** to route · Set **{name}** as start · Set **{name}** as destination · Delete **{name}** |
| `waypoint` (in route) | Remove **{name}** from route · Set **{name}** as start · Set **{name}** as destination · Delete **{name}** |
| `leg` | Insert waypoint here |
| `empty` | Add waypoint here · Route to here · Clear route |

Behavior: closes on action, `Escape`, or click-away (outside-click listener).
"Delete **{name}**" prompts a small inline confirm (reuses the existing delete
flow, `page.tsx` ~951, which removes from the catalogue and the route).

## 4. Map `contextmenu` handler + hit-testing

- **`Map.tsx`:** add `map.on('contextmenu', (e) => { e.preventDefault();
  onContextMenuRef.current?.({ lat: e.lngLat.lat, lon: e.lngLat.lng, point: e.point }); })`,
  mirroring the existing `onClick` ref pattern (line 148). `preventDefault()`
  suppresses the browser's native menu. Add an `onContextMenu?` prop.
- **`page.tsx`** provides the handler. It hit-tests with
  `map.queryRenderedFeatures(point, { layers: ['waypoints-dot', 'route-connector'] })`:
  1. A `waypoints-dot` feature hit → `{ kind: 'waypoint', waypoint, inRoute }`
     (resolve the waypoint by the feature's `id` property).
  2. Else a `route-connector` feature hit → `{ kind: 'leg', lat, lon, insertIndex }`.
  3. Else → `{ kind: 'empty', lat, lon }`.
  Then open `ChartContextMenu` at `point`.

## 5. Leg hit-testing → insert index (connector change)

Today `attachRouteConnector` draws the route as a **single** LineString, so a
`queryRenderedFeatures` hit can't tell which leg was clicked. Change it to emit
**one feature per segment** carrying `segIndex` (the index of the segment's first
point), mirroring `RoutePolyline.segments()`. Then a `leg` hit reads
`feature.properties.segIndex` and the insert index is `segIndex + 1` (insert the
new waypoint between the two endpoints of the clicked leg). Styling is unchanged
(dashed white); only the source geometry is split per-segment.

## 6. Waypoint creation (reuse existing)

"Add waypoint here" / "Route to here" / "Insert waypoint here" all create a saved
waypoint, reusing the existing path in `page.tsx`:

- Auto-name via the existing `nextWaypointName(existingNames)` helper.
- `POST /api/waypoints` with `{ name, lat, lon }`, append to `waypoints` state on success.
- Then apply the route mutator with the new id: `append` (Add here), `setEnd`
  (Route to here), or `insertAt(insertIndex, id)` (Insert here).

Factor the create-then-route step into a small async helper in `page.tsx`
(`createWaypointAt(lat, lon): Promise<string | null>`) returning the new id, so the
three menu actions share it.

## 7. Testing

Pure unit tests (`lib/route-plan.test.ts`, `lib/route-hit-test.test.ts`):

- Route mutators: `append` / `remove` / `insertAt` / `setStart` (present→moves to
  front; absent→prepends) / `setEnd` / `setIds` / `clear`, including derived
  start/end/via for 0, 1, 2, 3+ ids.
- Hit-target resolution: given a `queryRenderedFeatures`-shaped result, returns the
  correct discriminated `target` (waypoint > leg > empty precedence; `inRoute`
  computed from the current ids; `insertIndex` from `segIndex`).

Component (`ChartContextMenu`) + the map handler are verified by `npm run build
--workspace @g5000/web` + a manual chart smoke (no DOM test harness — #19).

## 8. Build / deploy notes

`packages/web` only — no engine, API, or schema changes (`/api/waypoints` POST,
PATCH, DELETE all already exist). `next build` is the type-level gate (stricter
than `tsc -b` for routes/components).
